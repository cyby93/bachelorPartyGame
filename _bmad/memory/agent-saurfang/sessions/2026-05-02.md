## Session 3 â€” HP upgrade on correct quiz answer

### Task
When a player answers the quiz correctly they now gain +10% of their class's default max HP (in addition to the skill upgrade they choose). E.g. Warrior (120 base) gains +12 HP per correct answer.

### Changes Made

#### server/entities/ServerPlayer.js
- Constructor: added `this.hpUpgrades = 0`
- New method `applyHpUpgrade()`:
  - Computes `bonus = Math.round(CLASSES[this.className].hp * 0.1)` (uses the class default, not current `baseMaxHp`, so the bonus is always flat)
  - Increments `baseMaxHp` and `maxHp` by `bonus`
  - Calls `this.heal(bonus)` so the player immediately gains the HP
  - Increments `this.hpUpgrades`
- `_snapshot()`: added `maxHp: this.maxHp` â€” was missing, so `maxHp` changes were invisible to the delta system
- `toDeltaDTO()`: added `if (cur.maxHp !== prev.maxHp) delta.maxHp = cur.maxHp`

#### server/GameServer.js
- `_resolveQuiz()`: replaced the "mark wrong players as upgrade-done" loop with a unified loop that calls `p.applyHpUpgrade()` for correct players and adds wrong players to `_quizUpgradesDone`. HP upgrade fires for all correct players at resolve time, before skill upgrade selection.

### Notes
- No client changes needed â€” `maxHp` delta propagates to the host renderer via the existing state delta mechanism.
- HP bar on the host will reflect the new max immediately after the quiz resolves.
- No UI indicator on the controller screen for the HP gain (controller only sees skill upgrade options). Can be added later if Cyby wants it.

## Session 4 â€” Enemy Sandbox: spawn buttons broken + enemy list extension

### Bugs Fixed

#### Bug 1: Sandbox immediately game-overs with no real players
`_checkAllDead()` fires every battle tick. With only host (or host + bots) connected, `livingCount = 0` and `players.size > 0` â†’ immediate `gameover` scene transition. The spawn buttons then silently fail the scene guard.

**Fix â€” `server/GameServer.js`:** Added `if (this.currentLevel?.debugSandbox) return` at the top of `_checkAllDead()`. Sandbox never triggers game-over.

Also split the combined guard in `_onDebugSpawnEnemy` and `_onDebugClearEnemies` into two separate checks, each emitting a `DEBUG_ACTION_RESULT` error so the sandbox status bar shows exactly why a spawn failed rather than silently returning.

#### Bug 2: Buttons destroyed mid-click by frequent innerHTML replacement
`renderSandboxPanel()` was called on every `STATE_DELTA` event (~20Ă—/sec). Each call replaced `sandboxActionsEl.innerHTML`, destroying all button DOM nodes. Clicks were silently lost if a delta arrived between mousedown and mouseup.

**Fix â€” `client/host/main.js`:**
- Removed `renderSandboxPanel()` from the `STATE_DELTA` handler entirely (panel state doesn't depend on delta data).
- Added `if (sandboxActionsEl.childElementCount === 0)` guard â€” buttons are built once per sandbox activation, not on every call.
- Added `sandboxActionsEl.innerHTML = ''` when `!activeSandbox` so buttons are properly cleared on scene exit.
