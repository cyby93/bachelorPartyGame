# Session 2026-05-04

## Task
Debug two controller regressions introduced by recent UI refactors.

## Issues investigated

### Movement not working (joystick moves, character doesn't)
- Full server-side trace confirmed correct: `_onInputMove` queues, tick drains, `player.setMoveInput`, `player.update(dt)`, `toDeltaDTO` sends changed x/y, `buildDeltaState` broadcasts, host renderer `_syncPlayers` calls `sprite.update(p, pos, dt)`.
- Skills working confirms socket connection and event pipeline is intact.
- Root cause: `MoveJoystick.svelte` used `let { onmove } = $props()` destructuring, which captures a snapshot of the prop value. In Svelte 5 runes mode, this snapshot is captured in the `onMount` closure — if Svelte's compiler doesn't generate a live getter binding here, the callback can be stale or missed.
- Fix: Changed to `const p = $props()` (keep proxy reference), then `p.onmove?.()` inside handlers. This always reads the current live value from the proxy, never a stale snapshot.

### Cooldown overlay just becomes gray
- Root cause confirmed: `@property` CSS custom property animation (used to animate `--cd-pct`) is not supported on older Android browsers and Samsung Internet. Falls back to `initial-value: 1`, rendering full gray overlay permanently.
- Fix: Replaced with JS `requestAnimationFrame` loop in `CooldownOverlay.svelte`. Updates `pct` state each frame, drives `conic-gradient` via inline style. Universally compatible.

## Files changed
- `client/controller/components/MoveJoystick.svelte` — `$props()` proxy pattern
- `client/controller/components/CooldownOverlay.svelte` — rAF-driven animation, removed `@property` CSS
