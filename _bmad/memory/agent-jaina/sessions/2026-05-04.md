## Session — Host Sidebar → Svelte Refactor

**What happened:** Migrated the entire gameplay sidebar and lobby player list out of plain JS DOM manipulation in main.js into reactive Svelte components. Build passes clean.

**Key outcomes:**
- Created `client/host/stores/gameState.js` — writable store with players, stats, boss, objectives, levelMeta, npcs
- Created `client/host/components/GameplaySidebar.svelte` — replaces renderGameplayRoster + renderMeterList + renderGameplaySidebar + renderLevelPanel + formatObjective; handles level info card, objective card, shade of Akama card (conditional), raid roster (2-col, HP fill), damage/healing meters (top-5 full + compact pills)
- Created `client/host/components/LobbyPlayerList.svelte` — replaces renderDOMPlayerList for the lobby player list
- Removed ~230 lines of keyed-update boilerplate from main.js
- main.js now calls `syncGameState()` (writes store) and `updateLobbyStartBtn()` (updates bot count + start btn disabled state) instead of the old render functions
- index.html: cleared static card content from `#gameplay-panel` and `#player-list` so Svelte mounts cleanly
- Components use Svelte 4 legacy mode ($: reactive declarations, $store auto-subscription) — no runes, no conflicts

**Observations:**
- `currentObjectives` added as a separate tracked variable in main.js because OBJECTIVE_UPDATE provides objectives that override meta.objectives — needed to pass to store independently
- `updateLobbyStartBtn()` kept separate from `syncGameState()` because it also manages startBtn.disabled — store doesn't own button state
- Shade of Akama card is now `{#if shadeVisible}` conditional in Svelte — the live shade update block in STATE_DELTA was eliminated entirely
- CLASSES import removed from main.js (now only needed in Svelte components)

**Follow-up:**
- Future sidebar cards (buff tracker, objective progress per room) = new .svelte file under components/, no main.js surgery needed
- `gameState` store is the expansion point — add fields there when new server data needs to reach the sidebar

## Session — Ability VFX asset research

**What happened:** Researched how ability VFX assets should be generated and used in RAID NIGHT, then inspected the existing PixelLab objects tied to fireball / pyroblast / shadow bolt. Confirmed which asset is actually in the repo today.

**Key outcomes:**
- Settled rule: use PixelLab object sprites for world-things (projectiles, traps, totems, pickups, small spawned props), not as the default for lightning beams, explosions, novas, aura fields, or burst energy
- Confirmed `public/assets/sprites/projectile_fireball.png` is byte-identical to PixelLab object `pyroblast` (`8d3a0ce0-06d5-4115-a98a-177734b4e0bc`)
- Identified newer fireball object in PixelLab: `b06f51e7-876f-447d-99fc-c673971acb5a`
- Identified shadow bolt objects: 1-dir `c76973f1-0040-4755-a569-6660cad6dbd2`, 8-dir `b4810779-c025-4b02-b064-4a0c08fdb9c0`
- Recommendation for future UI/VFX coordination: Fireball should prefer the lighter fireball object, Pyroblast can keep the heavier pyroblast object, Shadow Bolt fits 1-dir object art, Lightning Bolt should remain procedural/beam-first

**Observations:**
- Current host pipeline is already aligned with this: projectile body sprite + trail/particles in `ProjectileSprite.js`, burst/area effects in `VFXManager.js`
- Full animated object support would add complexity and is only worth considering for premium/heavy spells like Pyroblast

**Follow-up:**
- If VFX projectile art is revisited later, treat object sprites as the default asset class for physical spell tokens in the world
- If animated object support is ever implemented, scope it to selective hero spells instead of all projectile abilities
